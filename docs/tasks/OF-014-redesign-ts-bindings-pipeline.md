# OF-014 - Redesign the TypeScript bindings / type-generation pipeline

- **Severity:** Medium
- **Status:** Resolved (2026-05-13). Design pass + end-to-end spike of the option 1 + option 3 hybrid landed on `worktree-of-014-spike-option-3`. Productionization tracked separately in [OF-015](./OF-015-productionize-typescript-generation.md). See [Spike outcome](#spike-outcome-2026-05-13) below for what shipped and what's deferred.
- **Source:** Spawned from [OF-006](./OF-006-ts-bindings-fallback-warning.md) on 2026-05-12. The OF-006 warning makes the fallback observable; OF-014 is the underlying design fix.

## Spike outcome (2026-05-13)

A working spike of the option 1 + option 3 hybrid landed on branch
`worktree-of-014-spike-option-3`. Iron-log now builds clean (zero OF-006
fallback warnings) with `bindings.ts` fully populated: schema-known types
emitted directly from `EntityDef`, user-owned long-tail types emitted via
a generated specta side-car binary.

### What landed in ontogen

- `src/servers/generators/ts_bindings.rs` (new) - schema-known emitter
  (option 1 half). Walks `EntityDef.fields` and produces `export type` TS
  aliases for the entity + `Create{Entity}Input` + `Update{Entity}Input`
  via a bounded mapping over `FieldType`. Also exposes
  `referenced_ts_types`, `schema_known_names`, and `long_tail` helpers
  used to partition the import set.
- `src/servers/generators/ts_sidecar.rs` (new) - option 3 half. Generates
  a `__ontogen_ts_export.rs` source file inside the user's crate that
  calls `specta::TypeCollection::register_mut::<T>()` for each long-tail
  type and prints `specta_typescript::Typescript::export(&types)`
  (configured with `BigIntExportBehavior::Number` to handle `i64`/`u64`).
  Runs the binary via cargo with a distinct `CARGO_TARGET_DIR` to avoid
  the parent cargo's target-dir lock (rust-lang/cargo#8938) and appends
  stdout to `bindings.ts`.
- `src/servers/mod.rs::generate_transport` - wires both phases ahead of
  the existing client generators (`ts_client::generate`,
  `transport::generate`). Env-guarded by `ONTOGEN_TS_SIDECAR_INNER` so
  the inner cargo invocation, which re-runs the user's build.rs, does
  not recurse into the side-car block again. After both phases, the
  existing client generators see a fully-populated `bindings.ts` and
  emit zero `FallbackRecord`s.

### What landed in iron-log

- `src/schema/stats.rs` (new) - `WorkoutStats` long-tail type with
  `#[derive(specta::Type)]`. Iron-log had zero custom (non-CRUD) API
  endpoints prior to this, so the side-car path had nothing to exercise.
- `src/api/v1/stats.rs` (new) - `stats::workout` custom endpoint that
  returns `WorkoutStats`.
- `Cargo.toml` - `specta-typescript = "=0.0.10"` (the version pinned to
  `specta = "=2.0.0-rc.23"`, which iron-log already uses via Tauri).
- `src/bin/__ontogen_ts_export.rs` - generated by the build script, not
  hand-written. Productionization should ensure this is gitignored or
  emitted somewhere outside the source tree.

### Mechanical hurdles, with the choice made

- **Recursion**: outer cargo runs build.rs, ontogen invokes the inner
  cargo, inner cargo re-runs build.rs, ... forever. Broken via the
  `ONTOGEN_TS_SIDECAR_INNER=1` env var set on the inner invocation;
  ontogen short-circuits the side-car block when it sees it.
- **Target-dir lock**: outer cargo holds a lock on its target dir; the
  inner invocation needs its own. `CARGO_TARGET_DIR` set to
  `${OUT_DIR}/ontogen-ts-sidecar-target` on the inner `Command`.
- **Crate-isolation for type access**: ontogen runs in build-script
  context and cannot reach the user's types directly (they live in the
  user's crate, not its `[build-dependencies]`). The generated side-car
  binary lives inside the user's crate so it *can* import them via
  `${lib_name}::schema::WorkoutStats`. Lib name derived by parsing
  `Cargo.toml` `[lib].name` (or `[package].name` with `-` → `_`); the
  module path is derived by stripping `crate::` from
  `ServersConfig.types_import_path`.
- **specta v2 BigInt default**: `Typescript::default()` refuses to
  export `i64`/`u64` fields without explicit opt-in. The generated
  side-car uses `BigIntExportBehavior::Number` so long-tail types
  containing those fields don't fail the build.

### Verification

`cargo build` in `examples/iron-log/src-tauri/`:

### Trying this on your project

Provisional recipe - the API will change during productionization. Lives
here, not in user-facing docs, so don't link external consumers to it.

1. **Point `[build-dependencies]` at this branch.** In your consuming
   crate's `Cargo.toml`:
   ```toml
   [build-dependencies]
   ontogen = { git = "...", branch = "worktree-of-014-spike-option-3" }
   ```
   (Path dep also works if your checkout is local.)

2. **Add `specta-typescript` to `[dependencies]`.** Version is pinned to
   the same specta release your existing setup uses:
   ```toml
   [dependencies]
   specta = { version = "=2.0.0-rc.23", features = ["derive"] }
   specta-typescript = "=0.0.10"
   ```
   If your specta is on a different version, find the matching
   `specta-typescript` release - rc.24 → 0.0.11, rc.25 → 0.0.12.

3. **Make sure your `ServersConfig` populates `schema_entities`.** This
   is automatic if you use `Pipeline::new(...).servers(...)` - the
   pipeline copies `schema.entities` into `servers_config.schema_entities`
   when the latter is empty. If you call `gen_servers` directly,
   populate `schema_entities` manually from the parsed schema.

4. **Add `#[derive(specta::Type)]` to every user-owned type referenced
   by a custom (non-CRUD) API endpoint's return type or input type.**
   Entity types and ontogen-generated DTOs don't need this - they flow
   through the schema-known path. Only the "long tail" (types you wrote
   that aren't entities or DTOs) needs the derive.

   Example: if you have
   `pub async fn workout_stats(s: &Store) -> Result<WorkoutStats, _>`,
   then `WorkoutStats` needs the derive. If it has fields of other
   user-owned types, those need the derive too - specta walks them
   transitively, but each type still needs to opt in.

5. **Make sure `types_import_path` in your `ServersConfig` points at
   the module where the long-tail types live.** Spike-grade hardcode:
   the side-car uses `types_import_path` (with `crate::` stripped) as
   the module path inside your lib crate for *all* long-tail types.
   If your long-tail types are scattered across modules, this spike
   won't reach them - the productionization fix is in the punch-list
   below.

6. **Build.** `cargo build` should print zero `ontogen: type ... not
   found in bindings.ts` warnings. Check that `bindings.ts` (at the
   `bindings_path` you configured) now contains TS aliases for every
   entity, every Create/Update DTO, and every long-tail type.

7. **Gitignore the generated side-car.** Add to your `.gitignore`:
   ```
   src/bin/__ontogen_ts_export.rs
   ```
   The file is regenerated on every build that has long-tail types;
   committing it adds noise without value.

   **If the consumer is a Tauri app, also add a `.taurignore`** in the
   Cargo-manifest dir (next to `src/bin/`) with the same entry:
   ```
   # src-tauri/.taurignore
   src/bin/__ontogen_ts_export.rs
   ```
   tauri-cli's dev-mode file watcher (verified in 2.10.1
   `src/interface/rust.rs::build_ignore_matcher`) uses a custom
   `.taurignore` filename and does **not** honour standard `.gitignore`.
   Without this file, the side-car write at the end of the first
   `tauri dev` build trips the watcher and queues another `cargo run`,
   which presents as a rebuild loop on dev-server startup. The
   productionization story for this lives in
   [OF-019](./OF-019-document-side-car-tauri-watcher.md).

8. **If your CI runner is tight on disk** (default GitHub Actions
   runners ship ~14 GB free), gate ontogen's `.servers()` stage behind
   an env var in your `build.rs` and trust the committed generated
   files; the side-car compiles a second copy of every transitive
   dependency in its isolated `CARGO_TARGET_DIR` and doubles the cargo
   footprint. A consumer's working pattern: env-gate locally + a
   manual-dispatch "drift" CI job that re-runs the full pipeline on a
   freed-disk runner and `git diff --exit-code`s the generated paths.
   Sketch:

   ```rust
   // build.rs
   let skip_servers = std::env::var("YOUR_APP_SKIP_SERVER_CODEGEN").is_ok();
   let pipeline = Pipeline::new("src/schema")
       .seaorm(...).dtos(...).store(...).api(...);
   let pipeline = if skip_servers { pipeline } else { pipeline.servers(cfg) };
   pipeline.build()?;
   ```

   Productionization should provide a first-class config knob for this
   (tracked in [OF-019](./OF-019-document-side-car-tauri-watcher.md)).

9. **Retire any pre-existing manual `specta::export_ts` setup.** This
   spike makes `bindings_path` an **output**, not an input - on every
   build, ontogen overwrites that file with schema-known types and
   appends side-car output. Any hand-curated content there will be
   lost. If your consumer had a separate binary populating
   `bindings.ts`, delete it.

**Things to watch for:**

- **First build is slow.** The inner cargo invocation uses a distinct
  `CARGO_TARGET_DIR`, so it re-compiles every dep from scratch. Cache
  warms on subsequent builds; the cold-start cost is real.
- **`i64`/`u64` fields in long-tail types are emitted as TS `number`.**
  Spike hardcodes `BigIntExportBehavior::Number` to match the
  schema-known emitter's behavior. If your long-tail type needs the
  `bigint` TS type instead, this spike won't give you that.
- **The side-car file accumulates.** If a build removes the last
  long-tail type, the previously-generated `__ontogen_ts_export.rs`
  stays in your source tree (just unused). Delete it by hand or
  re-add a long-tail type to regenerate it.
- **Adding a long-tail type doesn't trigger a rebuild on its own.**
  No `cargo:rerun-if-changed` for long-tail type sources. Touch
  `build.rs` or clean to force a rerun.
- **Tauri consumers may see a rebuild loop without `.taurignore`.**
  See step 7 above. Symptom: `tauri dev` finishes its first build,
  app launches, then immediately rebuilds. tauri-cli's watcher tripped
  on the side-car write. Fix is the `.taurignore` entry; productionization
  in [OF-019](./OF-019-document-side-car-tauri-watcher.md).
- **`cargo run` becomes ambiguous after the side-car lands.** Cargo
  sees two `[[bin]]` targets (the app + `__ontogen_ts_export`) and
  errors with "could not determine which binary to run." Consumers must
  add `default-run = "<app-bin-name>"` to `[package]` in their
  Cargo.toml. Tracked in [OF-019](./OF-019-document-side-car-tauri-watcher.md).
- **CI disk pressure.** The inner cargo's separate `CARGO_TARGET_DIR`
  doubles the target footprint and exceeds the ~14 GB free on default
  GitHub Actions runners. See step 8 for the env-gate pattern.

### Known spike-grade shortcuts (productionization punch-list)

- **`Option<Option<T>>` rendering**: Update DTOs for already-optional
  entity fields render as `T | null | null` (valid TS, semantically
  equivalent to `T | null`, but cosmetically ugly). Fix: detect inner
  optionality before wrapping.
- **`types_module_path` derivation**: hardcoded as
  `strip_prefix("crate::")` over `types_import_path`. Assumes every
  long-tail type lives under that single module. Breaks if the user has
  types scattered across modules. Fix: track the source module of each
  long-tail type during parse.
- **BigInt config**: hardcoded `Number`. Should be plumbed through
  `ServersConfig` for users whose serializer supports `bigint`.
- **Doubled compile time**: the inner cargo build re-compiles every
  dependency from scratch in its isolated target dir on first build.
  Cache hit on subsequent builds, but the cold-start cost is real.
- **Inner build does redundant work**: the env guard skips the side-car
  block but still runs schema-known emission inside the inner build.
  Idempotent but wasteful. Fix: guard the whole `generate_transport`
  function (or `gen_servers`) when `ONTOGEN_TS_SIDECAR_INNER` is set.
- **No `cargo:rerun-if-changed` for long-tail types**: changes to
  user-owned long-tail type definitions won't trigger a side-car rerun
  via cargo's change detection.
- **Side-car file is never cleaned up**: if the long-tail set becomes
  empty after a change, `src/bin/__ontogen_ts_export.rs` is left behind
  in the user's source tree. Fix: delete the file when long-tail is
  empty (or move generation to `OUT_DIR` + a tiny stub).
- **`Cargo.toml` requires manual `specta-typescript` entry**: iron-log
  needed an explicit dep edit. Productionization should either document
  this in a guide page or have ontogen surface a clear error message
  when the dep is missing.

## Problem

Today the TS surface (`transport.ts` + the per-client variant) is generated
by ontogen, but the *type definitions* it imports come from a separate
`bindings.ts` file the user has to populate themselves with
`specta::export_ts` (or `ts-rs`). That separation is the root cause of the
silent-untyping foot­gun [OF-006](./OF-006-ts-bindings-fallback-warning.md)
patched over:

- The user has to set up a second binary that calls `specta::export_ts!`
  with every entity / DTO type listed by hand. Forgetting one entity is
  invisible until something at runtime throws because the call site is
  typed `Record<string, unknown>`.
- Adding a new entity is a multi-step ritual: write the entity, register
  it with specta, run the export binary, run the build, then check that
  the cargo warning list is empty. None of that is enforced.
- The fallback path has to exist in the TS emitter (`transport.rs:146-151`,
  `ts_client.rs:70-80`) precisely because we *can't* trust the contents of
  `bindings.ts`. The "fix" we shipped in OF-006 is a warning - it doesn't
  remove the failure mode, it just makes it loud.
- The Pumice integration (the source of OF-006) hit this on the very first
  entity. It's not a power-user pitfall.

## Location

The pieces involved span both ontogen and the consuming app:

- TS emitters that consult `bindings.ts`:
  - `src/servers/generators/transport.rs:74-181` (`generate`)
  - `src/servers/generators/ts_client.rs:22-211` (`generate`)
- Configuration surface:
  - `ClientGenerator::HttpTs { bindings_path }` and
    `ClientGenerator::HttpTauriIpcSplit { bindings_path }` in
    `src/servers/config.rs` (and re-exported from
    `src/servers/mod.rs:19-21`).
- Where bindings come from in real consumers: `examples/iron-log/...`'s
  `specta::export_ts` binary (the pattern Pumice copied), and the absent
  documentation about how to set that up - the OF-006 proposal originally
  asked for that doc, but writing it would lock in the workflow we want to
  replace.
- Type-collection helpers that drive what *should* appear in bindings:
  `src/servers/types.rs::collect_ts_import` and
  `src/servers/types.rs::collect_type_import` (the latter walks `syn::Type`
  for the Rust side; the former does string-based collection for TS).

## Current behavior

1. User configures `ClientGenerator::HttpTauriIpcSplit { output, bindings_path }`.
2. User stands up a separate `specta::export_ts` (or `ts-rs`) binary that
   knows about every entity / DTO they want exported.
3. Build runs. The TS emitter scans the *generated* TS and lists every
   type it references. For each referenced type it greps `bindings.ts` for
   `export type <Name>` / `export interface <Name>`.
4. Hits get added to an `import type { ... } from '<bindings>'`.
5. Misses get a placeholder + (post-OF-006) a cargo warning.

The user-visible failure modes:

- Missing `specta::export_ts!` registration → silent untyping (OF-006 turned
  this from silent to a warning).
- Bindings file written by a different generator (`ts-rs` vs `specta` vs
  hand-rolled) → identical user-facing brittleness because the contract is
  "export type X" string-grep.
- No story for transitive types: if entity `Workout` references `Exercise`,
  `Exercise` only appears in `bindings.ts` if the user listed it
  separately - the emitter doesn't tell specta what to export.

## Proposed direction (sketch, not a commitment)

Four plausible shapes; pick one in the design pass.

The schema-known surface is bigger than the original framing assumed:
ontogen already generates `Create{Entity}Input` / `Update{Entity}Input`
DTOs from `EntityDef` (`src/persistence/dto.rs`), and it parses every
entity type itself via `parse_schema`. So for entities and generated
DTOs, ontogen has full structural knowledge without consulting specta or
ts-rs at all - the `FieldType` enum is a complete description. The "we'd
have to write a Rust→TS emitter" cost from option 1 only bites for
genuinely user-owned types referenced by custom API endpoints (the long
tail).

1. **Ontogen owns the bindings.** Drive `specta` (or our own ts emitter)
   from the same `schema_entities` / scanned-API metadata the rest of the
   pipeline already has. `bindings_path` becomes an *output*, not an
   *input*. No fallback, no warning - if a type is referenced and isn't
   in the schema, that's a hard error. For the entity + DTO surface,
   "our own ts emitter" is a bounded mapping over `FieldType`; the open
   question is how to handle the long-tail user-owned types from custom
   endpoints (see option 3 for one answer).

2. **Ontogen tells the user what to register.** Keep the user's
   `specta::export_ts` binary, but emit a Rust file (or a build-script
   diagnostic) that lists exactly the types ontogen expects to find in
   `bindings.ts`. The user's export binary `include!()`s it. Misses become
   compile errors, not runtime placeholders.

3. **Ontogen generates the specta side-car.** ontogen already knows the
   exact list of type names the TS emitter referenced (`import_types`
   collection in `src/servers/generators/ts_client.rs:41-63` and
   `transport.rs`). Have ontogen *write* a small Rust binary into the
   user's crate that calls `specta::ts::export::<T>()` for each one, then
   `cargo run` it as a pipeline step and capture the output as
   `bindings.ts`. The user's only contribution is `#[derive(specta::Type)]`
   on types they want crossing the boundary. Missing derive becomes a
   *compile error in the generated binary*, not silent untyping.

   ontogen itself does not need to depend on specta - the dep lives in
   the user's crate (where it already lives in our example). The cost is
   real, though: the build now contains a compile-and-run-binary-in-the-
   user's-crate step (build orchestration, recursive `cargo`, target dir
   contention, error surfacing, rebuild-graph implications). Pairs
   naturally with option 1 - schema-known types emitted directly from
   `EntityDef`, the side-car only covers the user-owned long tail.

4. **Status quo + better docs.** Write the e2e guide the OF-006 proposal
   asked for. Cheapest, but doesn't address the underlying coupling - it
   just teaches the ritual. Worth noting that `examples/iron-log` itself
   doesn't have the side-car set up today (`bindings_path` resolves to a
   nonexistent file, so every type in the generated transport is a
   placeholder), so even the status-quo path needs work before it can be
   documented.

Option 1 is the most ambitious and probably the right end state. Option
3 is the highest-leverage incremental move - it closes the silent-untyping
hole without ontogen owning a TS type emitter, and the result is what
option 1 would converge toward for the long-tail surface anyway. Option
1 + option 3 pair cleanly: schema-known types from `EntityDef`, the rest
from the generated side-car, both flowing into a single `bindings.ts`
ontogen owns the path of. Option 2 is the lowest-effort closure of the
silent-failure mode if the side-car orchestration in option 3 turns out
to be too invasive.

## Effort

Large - this is a design discussion, not a one-PR fix. Expect:
- A short ADR / design doc covering the three options above.
- Prototype of whichever option wins, scoped to a single consumer
  (likely iron-log) before generalising.
- A migration plan for existing consumers (Pumice + iron-log) since any
  option that closes the fallback hole is a behaviour break for code that
  currently leans on `Record<string, unknown>`.

## Open questions

- Do we want ontogen to depend on `specta` directly, or stay generator-
  agnostic? (Affects option 1 directly. Option 3 sidesteps this - the
  specta dep lives in the user's crate, ontogen just writes a binary
  that uses it.)
- Is `bindings.ts` the only TS-side surface that needs this, or do we want
  the same treatment for any future client (Zod schemas, OpenAPI, ...)?
- Should the OF-006 warning be promoted to a hard error once OF-014 lands,
  or kept as a transitional belt-and-braces signal?

## Notes

- OF-006's warning text and `FallbackRecord` shape are deliberately
  user-facing now; whatever lands here should either remove the fallback
  entirely or repurpose the record into the new diagnostic.
- The OF-006 proposal's "documented e2e bindings path" half is *not*
  shipping under OF-006 because writing it would document the workflow
  this ticket wants to replace. If OF-014 stalls, reconsider — a stale
  guide is better than no guide.
