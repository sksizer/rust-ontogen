# ADR 0001 ŌĆö Markdown as a first-class store backend

## Status

**proposed** (2026-05-29)

## Context

Ontogen's pipeline is layered top-to-bottom as `schema ŌåÆ persistence ŌåÆ
store ŌåÆ api ŌåÆ servers ŌåÆ clients`, with each layer a code generator
producing Rust (or, on the leaf, TypeScript) under a versioned
`generated/` directory in the consumer crate. The persistence layer is
already pluralised: `src/persistence/seaorm/` emits SeaORM `Entity` +
`from_model` / `to_active_model` conversions, while
`src/persistence/markdown/` (1,430 LoC across `gen_writer.rs`,
`gen_parser.rs`, `gen_fs_ops.rs`) emits YAML-frontmatter file writers,
parser dispatchers, and filesystem-helper modules. The public surface
in `src/lib.rs` exposes both ŌĆö `gen_seaorm()` and `gen_markdown_io()`
are siblings.

The asymmetry is one layer up. `gen_store()` (`src/store/`) generates
the CRUD-with-hooks layer that downstream API and server generators
consume, and that layer is hardcoded to SeaORM:

| Coupling point | What's emitted |
|---|---|
| `src/store/mod.rs:116` | unconditional `use sea_orm::{ActiveModelTrait, EntityTrait, QuerySelect};` |
| `src/store/mod.rs:120` | conditional `use sea_orm::{ColumnTrait, QueryFilter};` (when filters appear) |
| `src/store/gen_crud.rs:347-365` | raw SQL via `sea_orm::Statement::from_sql_and_values` for has-many `set_parent` helpers, pinned to `DatabaseBackend::Sqlite` |
| `src/store/gen_crud.rs:368-385` | `fk_to_column_enum()` derives SeaORM `Column::ParentId`-style variants |
| `src/lib.rs:217` | `gen_store(entities, seaorm: Option<&SeaOrmOutput>, ...)` accepts the SeaORM metadata but the docstring acknowledges it is "reserved for future enrichment and currently has no effect on generated output" |

The markdown persistence layer therefore exists, but no generated code
above it can route to it. A consumer using `gen_markdown_io()` today
gets file-writing functions in a leaf module and nothing else ŌĆö no
store, no API, no servers ŌĆö because every layer above persistence
implicitly types its calls against the SeaORM-flavoured `Store` trait
the SeaORM-coupled `gen_store()` emits.

The trigger for this ADR is a recurring application class ŌĆö small-N,
human-editable, read-heavy datasets (knowledge bases, planning
trackers, configuration vaults, the project's own
`docs/planning/tasks/*.md` corpus) ŌĆö where a SQL database is the
wrong abstraction. For those, a markdown-and-YAML store is the
abstraction *and* the storage format simultaneously: editable in any
text editor, diff-friendly in git, indexable by Obsidian / grep /
ripgrep without an intermediate ORM. The persistence-layer markdown
generators were built with this case in mind; the rest of the
pipeline never followed.

This ADR decides what to lift, what to leave, and what the resulting
generator surface and runtime model look like.

## Decision

Adopt a **generation-time `Backend` choice** at the store layer:
`gen_store()` takes a `Backend::Seaorm { ŌĆ” }` or `Backend::Markdown {
ŌĆ” }` discriminant and emits one CRUD module per entity wired to that
backend's primitives. The output type (`StoreOutput` ŌĆö
`StoreMethodMeta`, `ScaffoldMeta`, `ChannelMeta`) stays unchanged so
downstream `gen_api()` / `gen_servers()` / `gen_clients()` are not
forced to branch. Two consumers running side-by-side in the same
workspace can therefore share `gen_api()` even though their stores
diverge.

The lift requires three structural changes:

1. **A `Backend` enum in `ontogen-core`** carrying per-backend
   metadata, replacing the current `seaorm: Option<&SeaOrmOutput>`
   parameter on `gen_store()`. `MarkdownIoOutput` (new) joins
   `SeaOrmOutput` as the backend-side artefact `gen_store()` consults
   when emitting the CRUD body.

2. **A `Store` trait contract that all backends satisfy.** The trait
   itself is *not* generated by ontogen ŌĆö it's expected to live in
   the consumer crate (or be generated once as a non-backend-specific
   `gen_store_trait()` step). Each generated `impl Store for
   {Backend}Store` block targets the same trait. Downstream API /
   server / client generators see a `Store` boundary, not a
   `SeaormStore`-shaped type.

3. **A small set of cross-backend semantic decisions** that the
   generator emits the same logical operation against, even though
   the implementations diverge sharply. See *Cross-backend semantic
   contract* below.

The SeaORM coupling identified in *Context* moves into a
`store::backends::seaorm` submodule under `src/store/`; markdown lands
as a sibling `store::backends::markdown`. The dispatcher in
`store::mod.rs` becomes a thin match on the `Backend` enum.

### Generator surface ŌĆö concrete entity example

Given a schema entity in `src/schema/`:

```rust
// src/schema/task.rs (consumer crate)
#[ontogen::entity(table = "tasks", route = "/api/tasks")]
pub struct Task {
    #[ontogen::id]
    pub id: String,
    pub title: String,
    pub status: String,
    #[ontogen::relation(belongs_to = "Epic", fk = "epic_id")]
    pub epic_id: Option<String>,
    #[ontogen::relation(many_to_many = "Tag", junction = "task_tags")]
    pub tags: Vec<String>,
}
```

**Today (SeaORM-only path):**

```rust
// build.rs
let schema = parse_schema(&SchemaConfig { schema_dir: "src/schema".into() })?;
let seaorm = gen_seaorm(&schema.entities, &SeaOrmConfig {
    entity_output:     "src/persistence/db/entities".into(),
    conversion_output: "src/persistence/db/conversions".into(),
    skip_conversions:  vec![],
})?;
let store = gen_store(&schema.entities, Some(&seaorm), &StoreConfig {
    output_dir:         "src/store/generated".into(),
    hooks_dir:          Some("src/store/hooks".into()),
    schema_module_path: ontogen::DEFAULT_SCHEMA_MODULE_PATH.into(),
})?;
gen_api(&schema.entities, &store, /* ... */)?;
gen_servers(&schema.entities, &api, &store, /* ... */)?;
```

**After this ADR (markdown path):**

```rust
let schema = parse_schema(&SchemaConfig { schema_dir: "src/schema".into() })?;
let md = gen_markdown_io(&schema.entities, &MarkdownIoConfig {
    output_dir: "src/persistence/markdown/generated".into(),
    vault_root: "docs/data".into(),                   // NEW: where the .md files live
    layout:     MarkdownLayout::PerEntityDir,         // NEW: docs/data/tasks/<id>.md vs flat
    id_strategy: IdStrategy::SlugFromTitle,           // NEW: how new records get filenames
})?;
let store = gen_store(&schema.entities, &StoreConfig {
    output_dir:         "src/store/generated".into(),
    hooks_dir:          Some("src/store/hooks".into()),
    schema_module_path: ontogen::DEFAULT_SCHEMA_MODULE_PATH.into(),
    backend:            Backend::Markdown(&md),       // ŌåÉ the discriminant
})?;
gen_api(&schema.entities, &store, /* ... */)?;
gen_servers(&schema.entities, &api, &store, /* ... */)?;
```

The `gen_api` / `gen_servers` / `gen_clients` calls are byte-identical
between paths. The only differences are below the store boundary.

**Emitted store call sites (per CRUD op, generated):**

| Op | SeaORM backend emits | Markdown backend emits |
|---|---|---|
| `list_tasks(filter, page)` | SeaORM `Task::find().filter(...).paginate(db, n)` | `walk_dir(vault/tasks/)` + parser + in-memory `.filter().skip().take()` |
| `get_task(id)` | `Task::find_by_id(id).one(db)` | `fs_ops::read_md(vault/tasks/<id>.md)` then `gen_parser::parse_task(...)` |
| `create_task(input)` | `task::ActiveModel { ... }.insert(db)` | `gen_writer::render_task(&task)` then `fs_ops::write_atomic(vault/tasks/<id>.md, body)` |
| `update_task(id, update)` | `task::ActiveModel { ... }.update(db)` | read ŌåÆ apply ŌåÆ re-render ŌåÆ atomic write-rename |
| `delete_task(id)` | `Task::delete_by_id(id).exec(db)` | `fs::remove_file(vault/tasks/<id>.md)` + cascade-link-cleanup if configured |
| `set_task_parent(child_id, parent_id)` | the raw-SQL escape hatch at `gen_crud.rs:347-365` | read child ŌåÆ mutate `epic_id` field ŌåÆ re-render ŌåÆ atomic write-rename |

The interesting case is `set_task_parent`: today, the SeaORM emitter
bypasses its own ActiveModel layer for FK-only updates and emits raw
SQL. The markdown emitter doesn't have a raw-SQL fast path ŌĆö the only
way to mutate one field of one record is the full read-render-rewrite
cycle. That's strictly more work per call, but the workload type this
backend is for (small-N, low-write-rate) makes it acceptable. The
`Backend` enum is where this divergence is allowed to surface.

### Relationship + query model

SQL's joins and indexes are not portable to a filesystem. The two
backends therefore agree on the *intent* of relationship operations
and diverge on the *implementation*. The contract is small enough to
enumerate:

| Intent | SeaORM | Markdown |
|---|---|---|
| `belongs_to` reference | foreign-key column on the table | YAML-frontmatter field carrying a wikilink: `epic_id: "[[E0042]]"` |
| `has_many` reverse-walk | indexed SELECT against child FK column | filesystem walk of the parent dir + parser filter (cost: O(N), but N is the per-folder count, not the global record count) |
| `many_to_many` | junction table joins | a YAML-frontmatter list on either or both ends: `tags: ["[[productivity]]", "[[ops]]"]`. Authoritative side is configurable per relation; the other side is a derived view. |
| Sorted list / pagination | SQL `ORDER BY ... LIMIT ... OFFSET` | parse-all ŌåÆ in-memory sort ŌåÆ skip/take. Acceptable up to ~10k records per entity at small page sizes; an explicit cap is part of the runtime config |
| Filter by field equality / range | indexed SQL WHERE | linear filter post-parse. **No emitted index** in this first revision (see *Out of scope*) |
| Atomicity of single-record write | DB transaction wraps the row | `tempfile::NamedTempFile` + `rename()` for atomic single-file replacement; the OS guarantees rename atomicity within a filesystem |
| Atomicity of multi-record write | DB transaction wraps the rows | **not supported**; emit a `store::AtomicityError` at runtime if multi-record mutation hits a partial-write failure. Consumers can adopt a write-journal pattern themselves if they need it (see *Consequences*) |

The wikilink convention is deliberate ŌĆö it matches Obsidian's own
syntax, which means the same files render as a navigable graph in
that tool without any extra rendering layer. `gen_markdown_io`'s
existing `wikilink()` / `yaml_inline_wikilink_list()` helpers at
`src/persistence/markdown/gen_writer.rs:107-115` already emit this
shape; the store layer adopts it as its citation format.

### Cross-backend semantic contract

The minimum the `Store` trait pins, so the generated API layer
doesn't need to know which backend it talks to:

1. **`id` is the primary key, always a `String`.** SeaORM allows
   integer or UUID; the markdown backend can't naturally use integers
   (they'd produce horrendous filenames). Forcing `id: String` across
   both means consumers using SeaORM today need to confirm their
   schemas use string IDs before opting into markdown. This is a
   real constraint, called out in *Consequences*.

2. **Single-record CRUD operations are atomic.** Multi-record
   operations are best-effort; either backend may emit a
   `BatchPartialFailure` error variant. (SeaORM could enforce
   transactional semantics; markdown can't; the trait doesn't promise
   what the weaker backend can't deliver.)

3. **`list()` returns records in some stable order**, but the order
   is unspecified at the trait level. Backends document their own
   default. Consumers needing a specific order pass an explicit
   `OrderBy` argument.

4. **Hook lifecycle (`before_create`, `after_create`, etc.) fires
   identically on both backends.** The hook surface stays exactly as
   `gen_hooks::scaffold_hooks` produces it today.

5. **The `gen_api()` / `gen_servers()` / `gen_clients()` outputs are
   byte-identical** between backends for the same schema. This is the
   load-bearing invariant: if a downstream pipeline change breaks it
   (e.g. a new SeaORM-specific argument leaks into the API forwarding
   function), that change needs an ADR or amendment here.

### Where SeaORM assumptions hide that must be lifted first

A pre-ADR-implementation refactor pass moves the SeaORM-specific code
out of the dispatcher path. The known sites:

| Site | Today | After lift |
|---|---|---|
| `src/store/mod.rs:116-120` | unconditional `use sea_orm::...` at file top | moved into `store::backends::seaorm::preamble()` |
| `src/store/gen_crud.rs:347-365` | inline raw-SQL `set_parent` helper | becomes `Backend::emit_set_parent_helper(code, entity, fk)`; SeaORM impl is the existing code, markdown impl is read-mutate-rewrite |
| `src/store/gen_crud.rs:368-385` | `fk_to_column_enum()` | SeaORM-only helper, moved under that backend's module |
| `src/lib.rs:217` | `gen_store(entities, seaorm: Option<&SeaOrmOutput>, ...)` | `gen_store(entities, &StoreConfig { backend: Backend::..., ... })`. The current `Option<&SeaOrmOutput>` is a partial pre-emption of this shape; deleting it is a small API break (the docstring already notes it has no current effect) |
| `crates/ontogen-core/src/ir.rs:33` | `SeaOrmOutput` is the only persistence-side output type | grow a sibling `MarkdownIoOutput`; `Backend` wraps either |

None of these lifts change generated behaviour for SeaORM consumers.
They are pure structural refactors gated by the existing snapshot
tests in `src/snapshots/`.

## Consequences

**Positive:**

- Small-N projects that don't want SQL get a native backend, not a
  workaround. `docs/planning/tasks/*.md` for SDLC-style tooling,
  Obsidian-vault-as-application-state, knowledge-base CRUD UIs ŌĆö
  these become straightforward consumers.
- The Backend boundary forces the downstream API / server / client
  generators to stay decoupled from persistence specifics. That is
  good architectural hygiene irrespective of whether more backends
  ever land (an in-memory backend for testing, a Redis backend, a
  Sled backend ŌĆö all become single-author additions once the seam
  exists).
- Markdown persistence's existing 1,430 LoC of writers and parsers
  earns its keep beyond the leaf use case; right now it is feature
  software that no upstream layer can route to.
- The decision is a generation-time choice, so binary size,
  dependency graph, and runtime memory of a consumer using only one
  backend are unaffected by the other backend's existence.

**Negative:**

- `id: String` constraint becomes binding. Existing SeaORM consumers
  using `i64` or `Uuid` primary-key columns need to migrate (or
  opt out of markdown forever). This affects the project's existing
  examples ŌĆö they would need an audit pass before the first markdown
  consumer ships.
- Multi-record atomicity is genuinely weaker on markdown. Consumers
  that need transactional batch operations must either stay on SeaORM
  or layer a write-journal on top of the markdown backend themselves.
  This is a documented limitation, not a bug to fix in the backend.
- Query performance is O(N) per list operation against the
  per-entity directory. For the use case the backend is aimed at (N
  in the low thousands per entity at the outside), this is fine;
  beyond that, consumers should not pick this backend. The ADR does
  not commit to ever building an emitted index for the markdown
  backend ŌĆö that is a separate decision deferred to a later ADR if
  the use case earns its keep.
- The `Backend` enum is closed (Rust enum). Out-of-tree backends
  require an upstream PR. This is an explicit choice ŌĆö the
  alternative (a `Backend` trait that downstream crates implement) is
  considered and rejected below.

**Follow-on work this implies (not committed to in this ADR; tracked
as ontogen tasks once accepted):**

- Backend abstraction lift PR (~the table of lifts above, no behaviour
  change for SeaORM).
- `gen_markdown_store` codegen ŌĆö the markdown side of `gen_store`.
- A pilot consumer crate exercising the end-to-end markdown pipeline,
  e.g. a tasks-as-markdown-files mini-app. The SDLC plugin's own
  `docs/planning/tasks/` corpus is the obvious dogfooding target if
  ontogen ever wanted to self-host its planning data.
- Documentation pass on `site/...` ŌĆö when to pick which backend, the
  scale ceiling for markdown, the `id: String` constraint, migration
  notes for SeaORM consumers wanting to evaluate markdown.

## Alternatives considered

### A. `Backend` as a runtime trait, polymorphic at construction time

Instead of choosing a backend at codegen time, generate both
`SeaormStore` and `MarkdownStore` and let the consumer pick at
runtime via `Box<dyn Store>` or generics. Rejected because:

- Doubles generated binary size and dependency footprint for any
  consumer that only ever uses one backend.
- Forces every generated method signature through trait objects
  (boxing, dyn dispatch) or through generic parameters that propagate
  to every call site in `gen_api()` and `gen_servers()`.
- Provides no operational benefit ŌĆö consumers do not, in practice,
  swap their persistence layer at runtime. The "switch backend with
  one config flag" story is satisfied equally well by a generation-
  time choice.

### B. Markdown as a SeaORM custom storage backend

SeaORM supports custom `DatabaseBackend` implementations. In theory
markdown could be hidden behind one, and `gen_store()` would not need
to know about it. Rejected because:

- The semantic gap is too wide: SeaORM assumes SQL semantics
  end-to-end (typed columns, deterministic query planner, join
  algebra), and a markdown "backend" emulating those would be more
  complex than a parallel store generator. The ORM tax is paid for
  semantics the user explicitly does not want.
- Hides the architectural reality. Consumers picking markdown
  *because they want plain files* would be funneled through a SQL
  ORM dialect they never asked for. That's the worst of both
  abstractions.
- Forecloses other genuinely-non-SQL backends (in-memory, Sled, JSON
  blobs in S3) ŌĆö they'd all have to pretend to be SeaORM. The
  `Backend` enum approach is general.

### C. `Backend` as an out-of-tree trait, downstream crates implement

Same generation-time choice, but the `Backend` enum becomes a trait
that consumer crates implement and pass into `gen_store()`. Rejected
for now because:

- The set of viable backends is small enough that closed-set
  enumeration is fine for the next year of work.
- A trait-based extension surface implies a stability contract on the
  generator's internal helpers (the `emit_*` methods backends call
  into). That contract is genuinely not ready ŌĆö gen_crud.rs is one
  refactor away from churning hard.
- If the closed-set approach turns out to be wrong, opening it later
  is straightforward ŌĆö convert the enum to a trait, default impls for
  the existing two variants. The reverse migration (closing an open
  trait) is much harder.

### D. Do nothing; keep `gen_markdown_io()` as a leaf-only utility

The path of least resistance. Rejected because:

- It leaves real software (the 1,430 LoC of markdown persistence) in
  a dead-end position ŌĆö written, tested, and shipped, but unreachable
  by the layered pipeline that is the rest of ontogen's value
  proposition.
- The use case is recurring enough (knowledge bases, planning
  trackers, the project's own SDLC corpus, every Obsidian-vault-
  shaped app) that punting it forces every consumer who hits the
  case to fork ontogen or hand-write the store layer.

## Notes

- The trigger for this ADR was a creative exploration prompt on
  2026-05-29 ŌĆö "what would supporting markdown-with-frontmatter as a
  backing datastore look like?" The discovery that
  `src/persistence/markdown/` already exists at 1,430 LoC reframed
  the question from "build it" to "wire it through the rest of the
  pipeline."
- The wikilink convention deliberately matches Obsidian's because
  the `docs/.obsidian/` folder at the project root signals that the
  maintainer is already an Obsidian user; aligning the generated
  citation format with that ecosystem is a small ergonomic win.
- Related: the SDLC plugin's task corpus at
  `docs/planning/tasks/*.md` is already YAML-frontmatter +
  markdown-body shaped. If ontogen ever wanted a dogfood-style pilot
  for the markdown backend, generating an `ontogen::entity` for the
  task schema and routing it through this pipeline would close a
  satisfying loop.
- The `id: String` constraint deserves its own checked-in design
  note before implementation begins. The constraint shows up only in
  the cross-backend trait contract, but it cascades into
  `gen_seaorm`'s column-type inference and into every existing
  example with `i64` IDs. An implementation task picking up this ADR
  should audit all in-tree consumers' ID types in its `## Today`
  table before proposing any non-additive changes.
- Revisit cadence: if no implementation work lands by the end of
  the next milestone, re-evaluate whether the recurring use case is
  real or whether SeaORM-only is fine for the projects this codebase
  actually serves. Architecture decisions made too far ahead of
  consumers tend to rot.
