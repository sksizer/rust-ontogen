# Pumice feedback round 2

Three new findings surfaced when Pumice upgraded to ontogen rev `168ff379`. The first round of feedback ([`2026-05-12-pumice.md`](./2026-05-12-pumice.md)) covered OF-001 through OF-012 and is fully resolved upstream as of the commits in that document's Resolved table.

The consumer's feedback log numbers these three findings as OF-013/OF-014/OF-015 in their own sequence. Upstream uses different IDs (OF-016/17/18) because the consumer's numbers collide with already-existing upstream tickets:

| Consumer ID | Upstream ID | Topic |
| --- | --- | --- |
| OF-013 (theirs) | [OF-016](../tasks/OF-016-classify-get-by-param-shape.md) | `get_*` classifier forces HTTP GET + `Path<String>` regardless of second-param shape |
| OF-014 (theirs) | [OF-017](../tasks/OF-017-param-import-substring-gate.md) | Type-import collector skips param wrapper structs whose name doesn't contain `Input`/`Query` |

Upstream's OF-013 / OF-014 / OF-015 — referenced from `tasks/README.md` — are unrelated to the consumer's numbering and remain as: AST-ified `param_to_owned_type` (resolved `71d76ce`), TS bindings redesign (resolved `c87ba64`), and TS pipeline productionization (in-flight).

## Findings

### Upstream [OF-016](../tasks/OF-016-classify-get-by-param-shape.md) (consumer's OF-013) — `get_*` classifier ignores param shape

**Severity:** Medium

**First hit:** Pumice task 0045, Bucket 4 — `export::get_filtered_sessions(state, filter: &ExportFilterRequest)` and `export::get_summary` both want a filter / sort body, but the classifier maps any `get_*` name to `CustomGet`, which routes as HTTP GET with the second param destructured as `Path(x): Path<String>`. The generated handler then tries to pass `&String` where the user fn expects `&ExportFilterRequest`, which fails to compile.

**Location:** `src/servers/classify.rs:46` (`if name.starts_with("get_") || params.is_empty() { CustomGet }`).

**Behaviour:** The name-prefix is the only signal — there's no way to say "this is a read-only POST" without renaming the function. Forcing GET also fixes the second param to be a single string id, which doesn't compose with body-carrying request structs.

**Workaround in use:** Renamed `get_filtered_sessions` → `filtered_sessions` and `get_summary` → `summary` so they fall through to `CustomPost`. The IPC behaviour is identical; the HTTP semantic loses the GET hint but gains a body.

**Proposed resolution:** Look at the second param's shape too. If it's a custom struct, emit `CustomPost` regardless of the `get_` prefix. Or: support an attribute like `#[ontogen::post]` on a `get_*` fn to opt out of the GET routing. See the upstream task for the full design and test matrix.

### Upstream [OF-017](../tasks/OF-017-param-import-substring-gate.md) (consumer's OF-014) — Type-import collector misses wrapper structs in `schema::*`

**Severity:** High

**First hit:** Pumice task 0045, Bucket 4 — defining `ExportRequest`, `ExportFilterRequest`, and `FilenameTemplateRequest` as `pub struct`s in `src-tauri/src/schema/mod.rs` (sibling to `ExportSummary`, `JournalSubmission`, `ProfileInput`, etc. which the collector picks up fine). The generated IPC / HTTP code referenced the wrapper names but didn't add `use crate::schema::ExportRequest;` etc. to the imports list, breaking compilation.

**Location:** `src/servers/generators/ipc.rs:117-127` and `src/servers/generators/http.rs:54-61`. Both wrap `collect_type_import` in a substring filter that only allows the walker to run when the rendered param type name contains `Input` or `Query`.

**Behaviour:** Inconsistent — `ExportSummary` (return-type only), `ProfileInput` (parameter-only, matches the substring), and `JournalSettings` (return + parameter) all get imported. The new wrapper structs in identical-shape parameter positions don't. Reproduced fresh on `168ff379`, not a stale-cache issue.

The AST walker fix in [OF-008](../tasks/OF-008-inner-type-strip-option.md) / [OF-010](../tasks/OF-010-collect-type-import-generics.md) went in, but the substring gate was left intact — so the walker is never invoked for params that don't match the heuristic. Same fix bucket as those two.

**Workaround in use:** Use fully-qualified `&crate::schema::ExportRequest` in the api/v1 fn signature. Per [OF-011](../tasks/OF-011-handler-arg-forwarding.md)'s AST-driven forwarding, ontogen preserves the fully-qualified path in the generated handler instead of emitting an import — so the type resolves without participating in the import collector.

**Empirical verification:** switching `export::run`'s signature from `&crate::schema::ExportRequest` to a bare `&ExportRequest` (with the type added to the local `use crate::schema::{...}` block) still fails compilation in the generated transport files because no `use` is emitted for those wrappers. Reverted to the fully-qualified path workaround.

**Proposed resolution:** Drop the substring filter. Walk every custom type appearing in any api/v1 fn signature (both parameter and return positions) and let the AST walker decide what to emit. See the upstream task for the unit-test matrix.

### Upstream [OF-018](../tasks/OF-018-ts-fallback-mistokenizes-generics.md) (consumer's OF-015) — TS bindings emitter mis-tokenizes generic return types

**Severity:** Low (cosmetic until OF-014 / specta sidecar lands; new finding from the upgrade attempt)

**First hit:** Upgrading to ontogen `168ff379`. While trying to drop the `NotificationPrefsMap` wrapper (OF-010 was supposed to fix the import side), the bare `Result<HashMap<String, NotificationPrefs>, AppError>` return triggered the new OF-006 fallback warning *and* generated an invalid TS placeholder:

```ts
// TODO: Type 'NotificationPrefs>' not yet exported from bindings.ts
type NotificationPrefs> = Record<string, unknown>;
```

The trailing `>` from `HashMap<…>` got captured into the placeholder type name, producing a TS parse error in `eslint`/`prettier`.

**Location:** `src/servers/generators/transport.rs:111-117` + `src/servers/types.rs:393` (`rust_type_to_ts`, string-prefix matcher) and `src/servers/types.rs:430` (`collect_ts_import`, splits only on `|` and trailing `[]`). Same area as [OF-006](../tasks/OF-006-ts-bindings-fallback-warning.md) (which only added the warning, not the parser).

**Behaviour:** `rust_type_to_ts` doesn't recognize `HashMap<K, V>` (or any other multi-arg generic). The string falls through unchanged, then `collect_ts_import` tokenizes badly and pushes a fragment containing `>`. The Rust-side import collector already does the AST-walk correctly post-OF-008/10; the TS emitter never received the same migration.

**Workaround in use:** Wrap the map in a named struct (`NotificationPrefsMap { entries: HashMap<…> }`) — the same OF-010 workaround. Restored after the upgrade attempt.

**Proposed resolution:** Walk the return-type AST when generating the placeholder, not the rendered string. The Rust-side import collector already does this correctly post-OF-008/OF-010; the TS emitter should mirror it.

Alternatively: once [OF-015](../tasks/OF-015-productionize-typescript-generation.md)'s productionization decides the OF-006 fallback path's fate (keep as belt-and-braces / promote to hard error / remove entirely), this bug may close naturally. See the upstream task's Direction A vs. Direction B discussion.

## Conventions

- One entry per finding. Don't bundle multiple proposals into one entry.
- Each entry must include: severity, first-hit context, location in ontogen source, behaviour, proposed resolution, and workaround if applicable.
- Severity reflects impact on Pumice's adoption of ontogen — not absolute importance of the issue to ontogen's user base.
- Consumer-side numbering and upstream-side numbering are tracked separately when they diverge. The crosswalk table at the top of this document maps between them.
