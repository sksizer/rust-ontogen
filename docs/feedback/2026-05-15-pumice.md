## Finding

### Consumer's OF-016 — seaorm entity emitter has no support for typed-enum columns

**Severity:** Medium

**First hit:** Pumice task 0032 — replacing three columns on `TimerSession` (`interval_kind`, `status`, `completion_kind`) from `String` to typed Rust enums (`IntervalKind`, `SessionStatus`, `CompletionKind`) end-to-end. The schema struct in `src-tauri/src/schema/timer_session.rs` declares the fields with their typed enums, and the enums derive `JsonSchema` / `specta::Type` so JSON/IPC/HTTP all round-trip cleanly. The ontogen-emitted entity file at `src-tauri/src/persistence/db/entities/generated/timer_session.rs`, however, declares the same columns as `pub field: String` — there is no path to opt into sea-orm's `DeriveActiveEnum` for these columns.

**Location:** `ontogen-core` → the seaorm entity-emission path. Likely near the column-type translation that decides between primitive types, `String`, and custom types when generating the `Model` struct fields.

**Behaviour observed:**

- Every column whose Rust type is an enum that serialises/deserialises as a string ends up as `pub field: String` in the generated entity. No introspection of the field type's `Serialize` / `Deserialize` shape, no annotation channel to opt in.
- Hand-adding a `*Db` mirror enum into the entity file is a non-starter: ontogen regenerates the entity file from scratch on every `cargo check` via the build script, and any local additions are clobbered.
- `seaorm_skip_entity` does not exist as an annotation. Only `seaorm_skip_conversions` is available.

**Workaround in use:**

- Pumice's schema annotation declares `seaorm_skip_conversions(["TimerSession"])` so ontogen continues to emit the entity (still useful — `Model`, `ActiveModel`, relations, etc.) but skips the conversion file.
- Pumice hand-writes `src-tauri/src/persistence/db/conversions/timer_session.rs` (~30 lines): reads strings off the entity model and parses them via `parse_interval_kind` / `parse_session_status` / `parse_completion_kind` (warn-and-default branches for unknown values — only reachable for corrupted rows, since the migration adds `CHECK` constraints), and writes typed enum values via `enum.as_str()` on the active-model side.
- Each new typed-enum column repeats the same dance.

The escape hatch is acceptable for one entity. It does not scale.

**Proposed resolution:** Three plausible designs, listed from "most explicit" to "most inferred":

1. **Schema-side attribute.** Add `#[ontogen(string_enum)]` on a schema struct field whose type is a Rust enum. The attribute carries the variant→string mapping either inline:
   ```rust
   #[ontogen(string_enum(
       Focus = "Focus",
       ShortBreak = "ShortBreak",
       LongBreak = "LongBreak"
   ))]
   pub interval_kind: IntervalKind,
   ```
   …or by deferring to a marker trait on the enum (see option 2). Ontogen emits a `*Db` mirror enum in the entity file with `DeriveActiveEnum` + `string_value` per variant, declares the entity column as the `*Db` enum, and emits `From<SchemaEnum> for *Db` / `From<*Db> for SchemaEnum` in the conversion file.

2. **Marker derive on the enum.** Add `#[derive(OntogenStringEnum)]` to the schema-side enum (in addition to its existing `Serialize` / `Deserialize` / `JsonSchema` / `specta::Type` derives). The derive macro emits associated consts (`STRING_VALUES: &[(&'static str, Self)]`) that ontogen can read at codegen time. Less intrusive in the schema struct (the field declaration stays clean); requires ontogen to be able to walk crate dependencies to find the derive.

3. **Inferred from `Serialize` / `Deserialize`.** Detect `#[serde(rename_all = "...")]` + plain unit variants on a `Serialize` / `Deserialize` enum and infer the variant→string mapping. Risk: ontogen has to parse serde's attribute output, which is brittle. Probably the wrong default — but could be a "best effort" fallback when neither (1) nor (2) is present.

The architectural constraint from Pumice's side: the schema-side enum must remain sea-orm-free. Pumice's `pumice::types` crate (where `IntervalKind` lives) explicitly disclaims any dependency on sea-orm, because the same enum types are used in browser-only HTTP contexts (the dual-transport architecture). So the `DeriveActiveEnum` must live on a separate `*Db` mirror enum in the entity file, not on the schema-side enum. Designs (1) and (2) both support that; design (3) would require ontogen to generate a mirror enum from the inferred metadata too.

**Workaround in source:**

- `src-tauri/src/persistence/db/conversions/timer_session.rs` (hand-written; the file ontogen would otherwise emit is suppressed via `seaorm_skip_conversions(["TimerSession"])`).
- Comment header on that file: `// DO NOT regenerate via ontogen until OF-016 (rust-ontogen) lands.`

**Pumice-side cleanup task:** Pumice files this as task `0049-ontogen-typed-enum-columns.md` — it gates on the upstream change, then deletes the hand-written conversion module and unsets `seaorm_skip_conversions(["TimerSession"])`.

**Status:** New finding from PR #202 (2026-05-15). Workaround is committed.

## Notes

- The CHECK constraints in the corresponding migration (`m20260515_000001_typed_enum_checks.rs`) enforce the variant set at the DB layer regardless of whether ontogen ever picks this up. The feature here is type-safety in Rust at the entity layer, not data-integrity.
- If a typed-enum field is `Option<EnumType>`, the design has to compose with the existing OF-008 / OF-010 `Option<T>` AST walker. None of Pumice's 0032 columns are optional, so this corner wasn't exercised.
- TS bindings: ontogen emits bare references to `IntervalKind` / `SessionStatus` / `CompletionKind` on DTO fields, but does not emit their *definitions* in `types.ts` (entity-field enum types aren't part of the "long tail" collected by the TS sidecar). Pumice currently appends union-literal TS aliases via a `build.rs` post-step. Worth folding into the same upstream feature so the TS side gets the enum unions for free — track as a sub-bullet of OF-016 or as a separate follow-up.
